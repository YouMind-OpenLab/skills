# Adaptation Playbook: Existing Content → Kit Broadcast

> Use when the user already has source material: blog post, long article, release notes, transcript, or another newsletter draft.

## Supported modes

| Mode | What it means |
|------|----------------|
| Cross-post | Adapt an existing article into a Kit-native broadcast |
| Condense | Turn long content into a shorter broadcast |
| Revive | Refresh an older Kit broadcast with new framing |
| Excerpt | Create a short feed/public teaser with one CTA |

## Step 1 — Diagnose the source

Classify the source:
- Long-form article
- Product release notes
- Founder/creator note
- Technical write-up
- Digest / list of links

Then determine:
- What is worth keeping?
- What is too bloggy / too slow / too SEO-shaped?
- What should become the CTA?

## Step 2 — Strip non-Kit baggage

Remove or rewrite:
- SEO intros
- multi-level heading sprawl
- heavy footnote structure
- long code blocks unless essential
- platform-specific widgets or embeds
- multiple competing CTAs

If the source was designed for search or archival reading, reorient it for inbox scanning.

## Step 3 — Rebuild the opening

The opening must feel like a live broadcast, not a repost.

Replace slow or generic intros with one of:
- "Quick update on..."
- "Three things changed..."
- "If you're following X, here's the part that matters..."
- "A short note before the details..."

## Step 4 — Reframe for Kit surfaces

Create:
- a **subject line** that can stand alone in the inbox
- **preview text** that gives a second reason to open
- a **first paragraph** that works both in email and on the public feed

If the content will be public, also think about thumbnail/SEO surface quality.

## Step 5 — Compress structure

Preferred end state:
- 2–4 sections
- short paragraphs
- one CTA

Common transformations:
- 8-section blog post → 3-section broadcast
- long conclusion → single closing paragraph + CTA
- deep tutorial → summary + link out
- verbose launch note → key changes + why + CTA

## Step 6 — Decide public/private behavior

| Situation | Suggested mode |
|-----------|-----------------|
| Personal/internal draft review | Private |
| Newsletter archive / creator profile content | Public |
| Not sure whether the user wants public web visibility | Ask |
| Public URL is important for sharing | Public, but warn that API URL return may be incomplete |

Private drafts should explicitly remind the user that Kit shows them in `https://app.kit.com/campaigns`.

## Step 7 — Self-critique

- [ ] Source no longer reads like a repost?
- [ ] Subject + preview + opening are aligned?
- [ ] Broadcast has one main action?
- [ ] Body is compact enough for inbox scanning?
- [ ] Public/private mode matches the user's intent?
- [ ] Public-feed presentation is acceptable if public?

## Step 8 — Output contract

Return:
- final title / subject used
- visibility mode
- best public URL if available
- fallback management path if not
- short conformance note explaining how the source was adapted for Kit
