# Adaptation Playbook: Existing Content → Beehiiv Post

> Use when the user already has source material: long-form article, previous newsletter, release notes, digest, report, or another platform's post.

## Supported modes

| Mode | What it means |
|------|----------------|
| Cross-post | Adapt an existing article into a Beehiiv-native post |
| Condense | Turn long content into a tighter Beehiiv edition |
| Revive | Refresh an older Beehiiv post with new framing or routing |
| Excerpt | Create a teaser, digest item, or premium-upgrade note |

## Step 1 — Diagnose the source

Classify the source:
- evergreen blog post
- launch note
- long-form essay
- technical write-up
- curated digest
- premium content teaser

Then determine:
- what is still the real thesis?
- what parts are too slow / too SEO-shaped / too web-only?
- what should the Beehiiv CTA become?
- what publication objective fits: acquire, retain, monetize, or recap?

## Step 2 — Strip non-Beehiiv baggage

Remove or rewrite:
- SEO lead paragraphs
- multi-level heading sprawl
- web-only sidebar references
- heavy embed / iframe assumptions
- multiple competing CTAs
- long code blocks unless essential
- platform-specific widgets

If the source came from WordPress or a docs site, rebuild it for email/web dual-surface reading.

## Step 3 — Rebuild the surrounding surfaces

Before rewriting the body, create:
- a **title** that works on the web archive
- a **subtitle** that sharpens the promise
- an **email subject** if the piece will send
- **preview text** that adds value, not duplication
- a **template choice** if the publication uses repeatable layouts

Beehiiv-native adaptation starts outside the body, not inside it.

## Step 4 — Rebuild the opening

Replace slow or search-shaped intros with something edition-native:
- "Here's what changed this week..."
- "If you've been following X, this is the part that matters..."
- "A quick operator note before the details..."
- "Today's release is simple: ..."

The opening should say why this issue exists now.

## Step 5 — Decide routing and visibility

| Situation | Suggested handling |
|-----------|--------------------|
| Utility article that should live on the site but not blast inboxes | Web-heavy or draft-first |
| Weekly newsletter issue | Both web + email |
| Premium insight / member note | Premium tiers only |
| Sponsor / housekeeping post | Consider hidden feed or explicit framing |
| Unsure about routing | Ask before publish |

Always decide:
- `web`, `email`, or `both`
- `all`, `free`, `premium`, or segmented
- feed visible or hidden
- template or no template

## Step 6 — Compress and reshape

Preferred end state:
- 2–5 sections
- quick opening
- one main CTA
- clean body for both email and browser

Common transformations:
- 10-section SEO post → 3-section edition
- release notes dump → what changed / why it matters / next step
- dense tutorial → key insight + one example + link out
- premium article → free teaser + upgrade CTA

## Step 7 — Self-critique

- [ ] Source no longer reads like a pasted cross-post?
- [ ] Title/subtitle/subject/preview are all doing distinct jobs?
- [ ] Audience routing is explicit?
- [ ] Feed visibility and template choice are intentional?
- [ ] Opening explains why the edition exists now?
- [ ] Body scans well in both inbox and browser?
- [ ] CTA is singular and publication-appropriate?

## Step 8 — Output contract

Return:
- final title used
- subtitle used
- surface plan: web | email | both
- audience / tier plan
- template choice
- best public URL if available
- short conformance note explaining how the source was adapted for Beehiiv
